# Simple RAG Demo

A production-structured Retrieval-Augmented Generation (RAG) demo built on the **RagQuAS** dataset — 201 hand-written Spanish question-answer pairs spanning ~30 domains (insurance, first aid, astronomy, recipes, travel claims, veterinary, yoga, etc.), created by the Instituto de Ingeniería del Conocimiento specifically to evaluate a complete RAG system.

The project demonstrates open-domain retrieval, measurable evaluation (retrieval recall@k against gold contexts, answer faithfulness against gold answers), and a clean Python package architecture ready to extend.

### Architecture decisions

These rules are enforced by convention and must be preserved as the codebase grows.

#### 1. `simple_rag/utils/` — generic, technology-level helpers only

Functions here must be reusable in any Python project without modification. They carry no knowledge of RagQuAS, Spanish text, chunking strategy, Pinecone index names, or any domain concept.

**Permitted imports:** standard library, third-party packages only.
**Forbidden imports:** `simple_rag.models`, `simple_rag.settings`, `simple_rag.data_pipeline`, `simple_rag.rag`.

If a function references a field name, a dataset, a prompt template, or a project setting by name, it does not belong in `utils/`.

| Module | Responsibility |
|---|---|
| `io_helpers.py` | Async file I/O, JSON serialization, SHA256 cache keys |
| `llm_helpers.py` | Gemini client setup, text generation (sync + async), LLM output parsing, tokenization |
| `pinecone_helpers.py` | Pinecone client, index management, embedding (sync + async), generic vector upsert |
| `text_helpers.py` | Generic string formatting utilities |

#### 2. `simple_rag/data_pipeline/` — ingest side only

Owns everything from raw dataset to indexed vectors: loading, exploding, deduplication, chunking, embedding, upserting.

**Permitted imports:** `simple_rag.utils`, `simple_rag.models`, `simple_rag.settings`.
**Forbidden imports:** anything from `simple_rag.rag`.

#### 3. `simple_rag/rag/` — query side only

Owns everything from user query to generated answer: routing, retrieval, reranking, generation, evaluation.

**Permitted imports:** `simple_rag.utils`, `simple_rag.models`, `simple_rag.settings`.
**Forbidden imports:** anything from `simple_rag.data_pipeline`.

#### 4. No cross-subsystem imports

`data_pipeline` and `rag` are independent subsystems that share infrastructure (`utils`, `models`, `settings`) but must never import from each other. A function needed by both belongs in `utils/`, not in either subsystem.

#### 5. `models.py` — single source of truth for data contracts

All Pydantic models live here. Neither `data_pipeline` nor `rag` defines its own models. The models file has no imports from anywhere in `simple_rag` other than standard library and Pydantic.

#### 6. `settings.py` — single source of truth for configuration

All tuneable values live here. Subsystems and their utils read from `settings`; they do not define their own constants for things that could change between runs. Shared utils (`simple_rag/utils/`) must not import `settings` — they accept configuration as function parameters instead, so the caller (pipeline or rag) decides which values to pass.

#### Dependency direction (summary)

```
simple_rag.data_pipeline  →  simple_rag.utils      ✓
simple_rag.data_pipeline  →  simple_rag.models     ✓
simple_rag.data_pipeline  →  simple_rag.settings   ✓
simple_rag.rag            →  simple_rag.utils      ✓
simple_rag.rag            →  simple_rag.models     ✓
simple_rag.rag            →  simple_rag.settings   ✓
simple_rag.data_pipeline  →  simple_rag.rag        ✗  NEVER
simple_rag.rag            →  simple_rag.data_pipeline  ✗  NEVER
simple_rag.utils          →  simple_rag.models     ✗  NEVER
simple_rag.utils          →  simple_rag.settings   ✗  NEVER
```

### Two subsystems

| Subsystem | Package | Purpose | When it runs |
|---|---|---|---|
| **Data pipeline** | `simple_rag.data_pipeline` | Transforms the raw dataset into a Pinecone vector index | Once (or on refresh) |
| **RAG system** | `simple_rag.rag` | Answers user queries using the index | At query time |

Shared code (`settings`, `models`, `utils/`) lives at the `simple_rag` level so both subsystems import from the same source of truth.

## Data flow

```
RagQuAS Parquet (201 rows, 19 cols)
        │
        ▼
 Step 1 — Load dataset → list[SourceRow]
        │
        ▼
 Step 2 — Explode text_1..5 per row → DocumentRecord per unique document
          Deduplicate by doc_id (SHA256 of text), keep question_1 variant first
          201 rows → 132 unique documents
        │
        ▼
 Step 3 — Chunk each document (≤300 tokens, 1-sentence overlap)
          → list[ChunkRecord]   saved to chunks-processed.parquet
          132 documents → 1063 chunks
        │
        ▼
 Step 4 — Embed (Pinecone Inference: multilingual-e5-large, 1024 dims)
          1063 embeddings generated
        │
        ▼
 Step 5 — Upsert to Pinecone index "ragchunks"
          metadata: topic, doc_id, link, source_domain, chunk_index, text
        │
        ▼
   Pinecone index "ragchunks" (1063 vectors, queryable)
```

### Pinecone vector schema

Each upserted vector carries this metadata (stored in Pinecone, no separate lookup needed at query time):

| Field | Type | Description |
|---|---|---|
| `text` | string | Chunk content — used to build the LLM context window |
| `topic` | string | Domain label (e.g. `seguros`, `yoga`) — primary filter for the RAG router |
| `doc_id` | string | SHA256 of source text — stable cross-run identifier |
| `chunk_index` | int | Position within parent document — enables adjacent-chunk expansion |
| `link` | string | Source URL — used for answer citations |
| `source_domain` | string | Parsed URL host (e.g. `airhelp.com`) — used for source attribution |

The vector `id` is `{doc_id}_chunk_{index}`.

### Retrieval strategy

The dataset spans 30+ domains with no semantic overlap (yoga, insurance, astronomy, recipes, veterinary…). The RAG router exploits the `topic` metadata to pre-filter the index before semantic search runs, narrowing the candidate pool from 1063 to ~30–50 topic-relevant chunks.

A typical retrieval flow:

```
User question
    │
    ▼
Router classifies query → topic label (one of 32 known topics, via Gemini)
    │
    ▼
Pinecone query with filter={"topic": "<detected_topic>"}
    │
    ▼
Top-k semantically similar chunks within that topic
    │
    ▼
Rerank: cap to max 2 chunks per doc_id (source diversity)
    │
    ▼
Gemini generates answer from retrieved chunks + cites link / source_domain
```

This three-lever approach — topic filter → semantic search → source-diversity reranking — improves precision without requiring any additional metadata beyond what the dataset already provides.

## Configuration

Copy `.env.example` to `.env` and fill in your keys:

```
GEMINI_API_KEY=...
PINECONE_API_KEY=...
```

All other settings are in `src/simple_rag/settings.py` and can be overridden via environment variables.

Key settings:

| Setting | Default | Description |
|---|---|---|
| `CHUNK_TARGET_TOKENS` | 300 | Target tokens per chunk |
| `CHUNK_OVERLAP_SENTENCES` | 1 | Sentence overlap between chunks |
| `CHUNK_MIN_SENTENCES` | 2 | Minimum sentences for a valid chunk |
| `DEFAULT_EMBEDDINGS_MODEL_NAME` | `multilingual-e5-large` | Pinecone Inference model (multilingual, 1024 dims) |
| `DEFAULT_EMBEDDING_DIMENSIONS` | 1024 | Embedding dimensions |
| `PINECONE_INDEX_CHUNKS` | `ragchunks` | Target Pinecone index (auto-created on first load) |
| `PINECONE_INFERENCE_BATCH_SIZE` | 30 | Texts per Pinecone embedding API call |
| `GEMINI_MODEL` | `models/gemini-2.5-flash-lite` | Model used for topic classification and answer generation |

## Running the pipeline

After `uv sync`, the `run-pipeline` CLI command is registered in your shell. It has three subcommands so each phase can be run and re-run independently.

### Subcommands

| Command | Steps | What it does |
|---|---|---|
| `run-pipeline preprocess` | 1–3 | Load dataset → explode/deduplicate → chunk → save `chunks-processed.parquet` |
| `run-pipeline load` | 4–5 | Read Parquet → embed (multilingual-e5-large) → upsert to Pinecone with metadata |
| `run-pipeline all` | 1–5 | Both phases in sequence |

```bash
# Typical first run
run-pipeline preprocess
run-pipeline load

# Or everything at once
run-pipeline all

# Re-index without re-chunking (e.g. after changing the Pinecone index)
run-pipeline load

# Test on a small subset before committing to the full dataset
run-pipeline preprocess --limit 20
run-pipeline load

# Built-in help
run-pipeline --help
run-pipeline preprocess --help
```

`preprocess` and `all` accept `--limit N` to cap the number of source rows — useful for iterating during development without processing all 201 rows.

`load` requires `chunks-processed.parquet` to exist (written by `preprocess`); it raises a clear error if it is missing.

Run from the project root. The `.env` file is read automatically.

The hand-off point between the two phases is `data_volume/assets/chunks-processed.parquet`. Inspect it between steps to verify chunking results before embedding.

## Data models

### `SourceRow`
Direct mapping of the 19-column RagQuAS Parquet schema. Four required fields (`topic`, `answer`, `question`, `variant`); all `text_i`, `context_i`, and `link_i` fields are optional because slots 4 and 5 are frequently empty.

### `DocumentRecord`
One unique source document after exploding and deduplicating `SourceRow`s. The `doc_id` is a SHA256 of the text content, making it stable and deterministic across runs. Multiple variant rows that reference the same text collapse into a single `DocumentRecord`.

### `ChunkRecord`
One chunk of a `DocumentRecord`, ready to upsert into Pinecone. Inherits `doc_id`, `topic`, `link`, and `source_domain` from its parent, and adds positional metadata (`chunk_index`) and size metadata (`char_length`, `token_count`).
